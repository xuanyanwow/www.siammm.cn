# Redis Mysql等主从数据同步延迟问题

1. 写完之后的读，主动选为读主库（需要同一网络请求内）
2. 二次读取——读不到从库的缓存，则再读一次主库（有较大风险，带来主库压力）
3. 关键业务读主，非关键业务读从——做分离



加锁，唯一索引，insert select where not exsit


## 无并发的情况

1. 先删除缓存，再更新数据库 -> 新请求触发 缓存空 再次缓存
2. 消息队列 + 异步重试 `把更新或者删除任务进入队列，保证Mysql和Redis的操作失败的时候可以再次重试`
3. 订阅binlog日志 订阅者更新缓存


## 高并发


1. 先删除缓存 再更新数据库 -> 设置缓存过期时间 + 延时双删
2. [建议 防止缓存穿透]先更新数据库 再更新缓存 -> 延迟消息 | 订阅binlog日志


## 强一致性

协议层面：分布式事务

业务层面：
- 消息暂存 `客户端进入队列 等待缓存更新结果好了 统一返回`
- 读写请求串行化`读写请求入队列，⼯作线程从队列中取任务来依次执⾏` (没有更新一半就有另外线程读取的可能)
- 使用Redis分布式读写锁 `将淘汰缓存与更新库表放入同⼀把写锁中，与其它读请求互斥，防⽌其间产⽣旧数据`
