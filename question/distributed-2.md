# 高可用架构

## CAP

- Consistence——一致性
- Availability——可用性
- Partition Tolerance——分区容错性

在一个`分布式系统`（指相互连接并共享数据的节点的集合中，如Mysql主从，而不是Redis集群这样子 节点无需互相连接的）中，当涉及`读写`操作时，只能保证CAP的两个，另外一个必须被牺牲

### C一致性

> v1: 所有节点在同一时刻都能看到相同的数据
>
> v2: 对某个指定的客户端来说，读操作保证能够返回最新的写操作结果

### A可用性

> v1: 每个请求都能得到成功或者失败的响应
>
> v2: 非故障的节点在合理的时间内返回合理的响应（不是错误和超时的响应）

### P分区容错性

> v1: 尽管出现消息丢失或分区错误，但系统能够继续运行
>
> v2: 当出现网络分区后，系统能够继续“履行职责”

放到分布式环境下来思考，我们必须选择P要素，因为网络本身无法做到100%可靠，有可能出现故障，所以分区是一个必然的现象。

如果我们的目标是CA，那么当发生分区时，为了保证C，系统需要禁止写入，当有写入请求时，系统返回error。而A又要求返回no error和no timeout。

`因此分布式系统理论上不可能选择CA架构，只能选择CP或者AP架构。`



### CP架构

N1数据已经更新，但是N1~N2之间的同步通道中断，为了保证数据一致性，客户端C来查询N2数据时候，N2需要返回给客户端`Error`，提示"系统发生了错误"

![image-20220210131428171](http://file.siammm.cn/picgo/image-20220210131428171.png)





### AP架构

N1~N2之间的同步通道中断，客户端C来查询N2数据时候，N2可以将自己拥有的数据返回给客户端，但不能保证数据一致性（最新数据）

![image-20220210131539643](http://file.siammm.cn/picgo/image-20220210131539643.png)





### CAP架构的细节

理论的优点清晰简洁，但缺点是高度抽象化，省略了很多细节，由于实践的各种复杂情况，可能出现误解和偏差，所以要注意细节



- CAP关注的粒度是数据，而不是整个系统
  - 系统中有很多数据，如商户数据、订单数据等等，不同数据会有不同的需求
- CAP是忽略网络延迟的
  - C在实践中是不可能完美实践的，数据复制过程中，节点之间的数据并不一致
  - 严苛场景下，例如和金钱相关的`用户余额、抢购的库存`，业务要求必须一致性，因此`单个`用户余额、`单个`商品库存，只能选择`CA架构`。例如，将用户分区的分布式架构分段或者Hash存储在不同节点。但如果节点故障，系统上有一部分用户使用不了（总比全部使用不了好）
- 如果没有发生系统分区，也就是P不存在的时候（节点网络连接一切正常） 我们应用C和A都保证
- 放弃并不等于什么都不做，需要为分区恢复后做准备
  - 例子：分区期间记录一些例子，当分区故障恢复后，系统根据日志进行数据恢复，使得重新达到CA状态
  - CP：节点连接中断后禁止故障节点查询，返回Error，但是非故障节点是可以使用的，记录状态，恢复后重新更新数据
  - AP：节点连接中断后，各自节点都可正常写入更新，系统恢复后自动同步，如果数据冲突，则提供报告，由人工选择

## ACID和BASE

CAP、ACID、BASE三者都是和数据一致性相关的理论

### ACID

`数据库管理系统`为了保证`事务的正确性`而提出的一个理论

- Atomicity原子性
- Consistency一致性
- Isolation隔离性
- Durability持久性

### BASE

是CAP理论中AP方案的延伸

- Basically Available 基本可用
- Soft State 软状态
- Eventually Consistency最终一致性



## FMEA

Failure mode and effects analysis 故障模式与影响分析

当我们设计出一个架构后，再使用FMEA对这个架构进行分析，看看架构是否存在某些可用性的隐患



在架构设计领域，FMEA的具体分析方法如下

1. 给出初始的架构设计图
2. 假设架构中某个部件发生故障
3. 分析此故障对系统功能造成的影响
4. 根据分析结果，判断架构是否需要进行优化

### FMEA分析表包含如下部分

1. 功能点：注意是从用户角度来看的功能，而不是系统各个模块功能点划分，如注册登录，而不是数据存储和Redis缓存
2. 故障模式：包含故障点和故障形式。例如`Mysql 响应时间达到3s`，不需要给出具体原因 如：磁盘坏道、慢查询、服务器到Mysql的连接网络故障、Mysql Bug等等；故障模式的描述尽量准确，多使用量化描述，避免使用泛式描述，例如`Mysql响应慢`
3. 故障影响：常见的影响有，也需要尽量准确
   1. 功能点偶尔不可用
   2. 功能点完全不可用
   3. 部分用户功能点不可用
   4. 功能点响应缓慢
   5. 功能点出错
4. 严重程度：一般分为`致命/高/中/低/无`
   1. 严重程度=功能点重要程度 * 故障影响范围 * 功能点受损程度
5. 故障原因
   1. 不同故障原因发生概率不同
   2. 不同的故障原因检测手段不一样
   3. 不同故障原因的处理措施不一样
6. 故障概率
7. 风险程度  = 严重程度 * 故障概率
8. 已有措施
   1. 检测告警
   2. 容错——系统能够通过备份手段应对，如Mysql主备机 业务服务器检测到主服务器不可用后，自动连接备机读取数据
   3. 自恢复——Hadoop检测到机器故障后，能够将存储在这台机器的副本重新分配到其他机器
9. 规避措施
10. 解决措施
    1. 为了解决密码暴力破解，限制密码重试次数限制
    2. 为了解决拖库导致数据泄露，将数据库中的敏感数据加密保存
    3. 为了解决非法访问，增加白名单控制
11. 后续规划

> 将FMEA分析表的后续规划整理出来，则得到下一步的架构演进方向，当前架构的不足

## 存储高可用

本质都是数据冗余，复杂点在于如何应对复制延迟和中断导致的数据不一致问题

从以下几个方面进行思考和分析

- 数据如何复制
- 各个节点的职责是什么
- 如何应对复制延迟
- 如何应对复制中断

### 常见的高可用存储架构

- 主备
  - 可使用✍️ mycat 做到主机的自动切换
  - 可使用✍️ Zookeeper来做主备倒换中介
- 主从
  - 主从跟主备的区别是，从机在普通时刻也需要帮忙干活（读请求）
  - 主备倒换架构 ：自动监听状态，自动决策切换
    - 互连式
      - 客户端只使用一个唯一地址，主机和从机需要共享一个地址，比如虚拟IP
      - 客户端记录主备机地址，哪个能访问就访问哪个；备机收到操作请求会拒绝
    - 中介式
      - 主备不互连，都去上报给中介，中介通知谁成为主机
      - 如果主机和中介断连，主机自动降级，等中介重新决策
    - 模拟式
      - 备机模拟成客户端向主机写读，探测状态  实现更加简单
- 主主
  - 不是所有场景都可以双向复制：用户ID自增长、商品库存等
  - 一般适用于临时性、可丢失、可覆盖的数据场景，例如 用户Session，用户行为日志，论坛的草稿数据等
- 集群
  - 数据集中集群
    - 适合数据量不大，集群机器数量不多的场景
  - 数据分散集群
    - 可伸缩性，适合业务数据量大，集群机器数量多的场景
- 分区



### 分布式事务算法

数据在集群中分开存储，就涉及到分布式事务，常用的算法

- ✍️ 2PC 二阶段提交算法
- ✍️ 3PC 三阶段提交算法



分布式事务算法的主要目的是为了保证`分散在多个节点上的数据统一提交或者回滚`



### 数据分区

一些自然灾害会导致整个城市或者地区的基础设施瘫痪，此时基于硬件故障的设计架构不可用。需要基于地理级别的故障来设计高可用架构

不同地区存储一部分数据，受影响的也只是一部分数据，故障恢复后，其他地区的备份数据也可以帮助故障地区快速恢复业务

## 业务高可用

### 异地多活

- 正常情况下，用户无论访问哪一个地点的业务系统，都能得到正确的业务服务
- 某地系统异常情况下，用户访问到其他地方正常的业务系统，也能够得到正确的业务服务

代价

- 系统复杂度会发生质的变化，需要设计复杂的异地多活架构
- 成本会上升，毕竟要多在一个或多个机房搭建一套业务系统



#### 同城异区

两个机房距离大约几十千米，可以搭建高速网络，可以实现在同机房几乎一样的传输速度，设计降低了复杂度

#### 跨城异地

距离最好要远一些，否则还不如采用同城异区方案

但带来架构的复杂度：光速真空传播每秒30万千米，光纤传输速度大约是20万千米，加上中间的网络设备处理，远远达不到光速的速度

中间也有其他风险，光纤被挖断等



正常情况下RTT  广州到北京大约是50ms，但遇到网络波动可能飙升至500ms甚至1s  

更不用说经常发生的线路丢包问题



延迟无法避免，又要保证无论哪个机房都业务正常，其中的重点还是在"数据"上，即`根据数据的特征来做不同的架构`



- 强一致性要求的数据，如余额等，实际是无法做到跨城异地多活的。 [书P166页]
- 数据不怎么改变，即使数据丢失也影响不大，如用户登录、新闻类网站、博客类网站等



>  设计技巧

技巧一：保证核心业务的异地多活

如用户注册登录，用户从A节点注册，马上到B节点注册，B节点此时还没有判断到A注册过了  允许注册

10万用户的项目，每天新增注册可能几千，修改信息可能几千，登录和使用业务才是最重要的，只做登录的多活即可

每个节点都有缓存用户信息，可以达到登录多活的目的



技巧二：核心数据最终一致性

数据同步是异地多活架构的`设计核心`



- 尽量减少数据同步，只同步核心业务相关的数据
- 保证最终一致性，不保证实时一致性



技巧三：采用多种手段同步数据

极端情况下，存储系统本身自带的同步功能可能难以满足业务



例如：Mysql5.1版本的复制是单线程复制，在网络抖动或大量数据同步时，延迟较长，



可采用辅助方案

- 消息队列方案
- 二次读取方式
- 存储系统同步方式
- 回源读取方式
- 重新生成数据方式



技巧四：只保证绝大部分用户的异地多活



假设账号是在北京银行开的，存钱、转账申请则一定要北京机房处理，上海机房不处理。 保证数据同源，就不会有同步不及时出现的漏洞

> 小明卡里有一万，先在北京机房申请转账，马上又到上海机房申请转账。由于上海机房还没同步到转账申请，余额没有扣除，转账成功。
>
> 虽然用户不会有这种操作，有也是自己程序员开发的bug，但是如果被黑客利用，则后果不堪设想



银行实时转账很难做到真正的异地多活，可以提供“转账申请”服务，上海机房记录请求，然后后台异步发起真正的转账操作，如果此时北京业务中心不可用，则请求继续等待重试





#### 跨国异地

延迟一般几秒，所以这里的多活跟跨城异地的多活，含义不完全一致。

主要应用场景

- 为不同地区用户提供服务
- 只读类业务做多活

#### 设计步骤 [书P173页]

1. 业务分级——按一定标准将业务分级，挑选核心业务，只为核心业务设计异地多活
   1. 访问量大小
   2. 核心业务
   3. 产生大量收入的业务
2. 数据分类——识别所有数据及数据特征，会影响方案设计
   1. 数据量（包括数据量和新增删除修改的可能量，新增删除修改数量越多，则要同步的越多，延迟概率越高，需要考虑相对应方案）
   2. 唯一性（是否要求多个异地机房产生的同类数据必须保证唯一，如订单号 ）只能一个中心点产生数据，要么需要设计一个数据唯一生成算法
   3. 实时性
   4. 可丢失性
   5. 可恢复性

![image-20220211151343285](http://file.siammm.cn/picgo/image-20220211151343285.png)

3. 数据同步方案
   1. 存储系统同步——如Mysql自带主从同步，主主数据同步
   2. 消息队列同步
   3. 重复生成

![image-20220211151439351](http://file.siammm.cn/picgo/image-20220211151439351.png)

4. 异常处理
   1. 在同步延迟、数据丢失、数据不一致的情况下，系统采取什么措施来应对
   2. 目的
      1. 避免少量数据异常导致整体业务不可用
      2. 问题恢复后，异常数据进行修正
      3. 对用户进行安抚，弥补用户损失
   3. 方案
      1. 多通道同步
         1. 一般两通道即可；不能采用相同的网络连接，不然网络故障 也都不可用；需要数据是可以重复覆盖的
      2. 同步和访问结合
         1. 不能采用相同的网络连接，不然网络故障 也都不可用；数据有路由规则，可以判断应该访问哪个机房的接口来读取数据；优先读取本地数据，本地数据读取不到就通过服务接口去取数据，大大降低跨机房的异地接口访问数量，适合于实时性要求非常高的数据
      3. 日志记录——用户恢复和分析
      4. 用户补偿

### 接口级的故障应对方案

例如

- 业务响应缓慢
- 大量访问超时
- 大量访问出现异常



一般原因可能性

- 内部原因——程序bug 死循环，接口导致数据库慢查询，程序逻辑不完善耗尽内存等
- 外部原因——黑客攻击，促销抢购等涌入大量用户，第三方系统大量请求，第三方系统响应缓慢等



#### 降级

提供部分功能，或者完全停止服务功能。如论坛只能看帖子，不能发帖子；APP日志接口完全停掉；

保证核心业务



- 系统后门降级——系统预留了后门用于降级操作，如一个URL，访问后执行降级指令
- 独立降级系统——批量操作，鉴权系统等等

#### 熔断

降级是用于自身系统的故障

熔断是应对依赖的外部系统故障的情况



- 熔断机制的关键是需要有一个统一的API调用层，由API调用层来进行采样或者统计，如果接口调用散落在代码各处就没办法进行统一处理了
- 另一个关键是阈值的设计，实践中一般先根据分析确定阈值，上线观察效果，再进行调优

![image-20220211152640137](http://file.siammm.cn/picgo/image-20220211152640137.png)





#### 限流

- 基于请求限流——适合业务比较简单的系统，如负载均衡系统、网关系统、抢购系统等
  - 限制总量
  - 限制时间量
- 基于资源限流



#### 排队

限流的一个变种，限流是直接拒绝用户，排队是让用户等待很长时间。如12306。调度模块需要检测服务模块的资源性能，如果空闲，就从排队中取出  发出真实的请求
