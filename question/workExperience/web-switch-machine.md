# web服务切换机房

问题背景可能为：

- 服务器过期，想用新人优惠买一个新服务器
- 业务规模大，线下机房迁移为云机房


## 自顶向下的迁移方向

常见web应用：

客户->域名->web server -> 实例services -> db 
                                |
                                -> cache

### 域名解析迁移 

自顶向下的意思就是先迁移域名（web server 和实例都在新机器准备好）但是此时`新机器还是连接旧的数据库和cache` 

> 注意 如果不可容忍迁移过程带来的高延迟 则需要机房之间搭建专线（延迟下降至几ms）

如果可以的话，在域名解析这里可以按地区或者比例  迁移10%流量到新机器IP  不要一次性全部解析到新机器

### cache进行迁移

等到流量全部完成迁移，进行cache的迁移

流程为：在新机器搭建好新的cache空缓存实例

业务层的cache`连接信息填写的是域名`，运维在新机器把域名DNS解析改为新机器IP，services重启即可完成cache的迁移

> 注意 需要在流量少的时候进行cache迁移，否则迁移后马上会带来cache miss的缓存透传问题

如果业务不能允许cache miss的问题，则需要先把cache改造成`双写缓存`或者缓存使用`主从同步`的架构，然后连接信息切换到新cache


### DB进行迁移

第一，得保证数据库同步完成，才能切流量，但数据同步总是有迟延的，旧机房一直在不停的写如数据，何时才算同步完成呢？

第二，只有域名和端口不发生变化，才能不修改配置完成切换，但如果域名和端口（主要是端口）发生变化，是做不到不修改配置和重启的。举个例子，假设原有数据库实例端口用了5858，很吉利，而阿里云要求你使用3200，就必须改端口重启。


所以，我们最终的迁移方案，是DBA在旧机房的数据库设置一个read only，停止数据的写入，在秒级别，RDS同步完成之后，业务线修改数据库端口，重启连接新机房的数据库，完成数据层的切换
