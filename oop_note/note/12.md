# 抽象工厂模式


> 提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类


每种产品有不同的实现（两大类）

客户端 -> 抽象工厂类 -> 抽象产品类

- 抽象工厂实现两大类的具体工厂   返回两大类产品类
- 产品类里实现不同逻辑
- 需要更改系统时候 只需要更改调用的工厂


例子
- DB 系统迁移


## 优缺点

- 具体的创建实例过程与客户端分离，底层是用Mysql还是Access  客户端无需在意，面向抽象接口编程
- 迁移工作量更低
- 缺点：声明的类更多了，客户端直接调用的地方多了的话 也需要改动很多才能迁移


## 优化升级

用简单工厂改进抽象工厂

- 简单工厂中 通过配置(在代码中 或者 配置文件) 配置一次，在调用create()方法的时候，就可以返回对应类型的类(返回类型声明是抽象接口类,实际是具体产品类)
- 缺点：增加大类别的时候，还是要在简单工厂中加入case代码，返回对应分类下的具体产品


## 优化升级

反射+抽象工厂   （依赖注入）

```php
public function createUser(): IUserModel
{
    // $type是配置文件中的值
    
    // 原本
    // if ($type == 'mysql') return new MysqlUserModel();
    // if ($type == 'access') return new AccessUserModel();
    
    $className = $type."UserModel";
    // 不固定了 可以在配置文件中任意扩展  只要类真实存在就可以   比如$type = 'Sqllite';
    
    return new $className;
}
```


## 附属笔记

- 简单工厂就是 在代码根据传递的type，用代码if else,case 返回具体的产品;
- 工厂模式就是 工厂抽象类-> A工厂 B工厂 -> 具体产品 `A工厂::create();` 里面是直接`new User()`
- 抽象工厂就是 抽象工厂类-> 大类产品抽象类 -> 具体产品(是多个大类的工厂 然后再是生产具体产品的工厂) `DataFactory::createUser();` `根据类型 MysqlDataFactory::createUser()`, `return new MysqlUserModel();`
- 在所有用到简单工厂的地方，都可以考虑用`反射+配置文件`来消除分支判断带来的耦合
- PHP语言优势：可以` new $className;` JAVA需要用反射类来实现，比较复杂
