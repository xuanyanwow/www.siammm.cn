第二节 策略模式

> 面向对象的编程，不是类越多越好，类的划分是为了封装，但分类的基础是`抽象`，具有相同属性和功能的对象的抽象集合才是类。

满200减10  满300减50， 代表的是同一个逻辑行为，只需要封装 两个传参初始化

其实在第一节，把不同算法封装成不同类，比如加减算法，就是一种基本的策略模式（拆分成两个策略），策略模式封装了变化。

- 便于维护
- 便于编写单元测试，每个逻辑有自己的类，可以单独编写测试


## 算法经常变动、扩展

引出`策略模式 Strategy`：定义了算法家族，它们之间可与互相替换，让算法的变化，不会影响到使用算法的客户。

简单理解：抽象类和封装类不用变化，在上层增加一个Cotext 上下文类，并可以在上下文中实现简单工厂的功能

### 简单工厂模式的代码

```php
DoClass $doer = Factory::create("下单");// DoClass是具体的算法执行者（抽象）
$doer->get();
```

### 策略模式的代码

```php
$doer = new DoerContext("下单");
$doer->get();
```

客户端只需要知道一个类及其功能，所以降低了`耦合`（客户端需要知道的越多 ，耦合则越高）

但其实在PHP下，结合IDE语法提示，可以看做无差别（PHP不是强类型，可以不声明$doer的类型）

#### 总结

在基本的策略模式中，客户端switch决定策略类。
在结合了简单工厂的策略模式中，具体的调用策略逻辑也由Context承担。
