## 单一职责

> 就一个类而言，应该仅有一个引起他变化的原因

职责越多，越容易引起变化，也就会造成复用可能性的下降。

职责耦合在一起，导致脆弱的设计。

## 开放 - 封闭 原则（面向对象设计的核心所在）

> 软件实体（类、模块、函数等）应该可以扩展，但是不可修改

设计出可以不断推出新版本新修改的软件，相对稳定。预留升级维护空间。

增加功能时，发现需要更改到原来的类，那么此时就要考虑在`小改动`的情况下，重构软件设计，`创建抽象`来隔离以后发生的同类变化。

此时不更改重构，那么在越复杂的情况下，越难以重构。

```
原有: 客户端->加法类
改善：客户端->运算类[-> 加法类 -> 减法类 -> 乘法类]
```

## 依赖倒转原则（+ 里氏代换原则）

易插拔、强内聚松耦合

> 针对接口编程，不要对实现编程（这样子才能重新写一个类来替换原有逻辑）

- 高层模块不应该依赖底层模块，两个都应该依赖抽象
- 抽象不应该依赖细节，细节应该依赖抽象。
- 
### 里氏代换原则

一个软件实体如果使用的是一个父类的话，那么一定适用于其子类，而且它察觉不出父类对象和子类对象的区别。

> 子类必须可以完全替换它们的父类 而不崩溃（但是可以更改具体逻辑，比如Oracle存储子类可以替换Mysql存储子类）


子类和父类之间，谁也不依赖谁，只是按照约定的接口运行。
