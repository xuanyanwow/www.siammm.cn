# 外观模式（门面模式）

买股票，耦合高，客户端压力大；

买基金，客户端只需要对接一个类，调用基金的买入卖出即可。

> 为子系统中的一组接口提供一个一致的页面，此模式定义了一个高层接口，这个接口使得这一子系统更加容易使用。

完美体现了`依赖倒转原则`和`迪米特法则`的思想，是最常用的模式之一。

## Facade 外观类

知道哪些子系统负责处理哪部分的职责，将客户的请求代理给适当的子系统对象。

## SubSystem Class 子系统具体类

子类中不需要知道Facade的任何信息和引用，对他们而言，Facade是他们的客户端


## 何时使用

- 设计初期阶段，应该有意识地将不同的两个层分离。比如经典三层架构，在数据访问层、业务逻辑层、表示层之间建立外观Facade。
- 在开发阶段，子系统往往不同的重构演化而越来越复杂，大多数模式会产生很多小类，这本是好事，但也给外部调用它们的用户程序带来了使用上的困难，增加外观Facade可以提供一个简单的接口，减少它们之间的依赖。
- 在维护一个遗留的大型系统时，可能这个系统已经非常难以维护和扩展了，但因为它包含了非常重要的功能，新的需求开发必须依赖于它，此时外观模式提供设计粗糙或者高度复杂的遗留代码的比较清晰简单的接口，让新系统与Facade对象交互，Facade与遗留代码交互所有复杂的工作
