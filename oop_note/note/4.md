# 装饰模式Decorator

动态地给一个对象添加一些额外的职责，就增加功能而言，装饰模式比生成子类更为灵活。

> 思考：可以用在功能增加上，比如正常结算订单settle()  在会员模式下可以注入VipDecorator 执行里面的settle() 增加会员积分
> 
> 思考：可以用在数据流分析中，按一定顺序一层一层流转统计
> 
> 思考：酒吧系统 点餐单冲账（主类逻辑） 更改点餐单状态后，可以加入 退款装饰 执行结算方式中 需要退款的逻辑。比如主管还想及时知道哪个员工操作了冲账，还可以加入微信推送装饰


## 总结

- 为已有功能动态地添加更多的功能
- 系统增加需求，一般是原有需求上装饰更多功能（比如一开始的用户结算提现，只有普通的转账逻辑，但是后面增加了用户风险评估的系统）
- 有效地把主类的核心功能和装饰功能区分开，可以取出相关类中重复的装饰逻辑


```php
// 对象接口
interface SettleComponent{
    public function settle($user, $order);
}

// 具体逻辑对象
class UserSettleComponent implements SettleComponent{
    public functio settle($user, $order)
    {
        echo "$user->user_name 结算了订单 $order->order_sn";
    }
}


// 装饰接口
abstract SettleDecorator implements SettleComponent{

    protected $component;
    
    
    public fucntion setComponent(SettleComponent $component){
        $this->component = $component;
    }
    
    public function settle($user, $order){
        if (!$this->component){
            $this->component->settle($user, $order);
        }
    }
}

// 会员逻辑 具体装饰类
class VipSettleDecorator implements SettleDecorator{
    public functio settle($user, $order)
    {
        parent::settle($user, $order);
        if (用户是会员){
            echo "$user->user_name 是一个会员，还需要执行增加积分的逻辑，增加的数量是 $order->price / 100";
        }
    }
}

// 高风险用户 具体装饰类
class HightRateSettleDecorator implements SettleDecorator{
    public functio settle($user, $order)
    {
        if (用户风控检测通过){
            parent::settle($user, $order);
        }else{
            echo "$user->user_name 是一个高风险用户，订单状态强制进入待审核状态，而不是普通自动结算状态";
        }
    }
}
```

客户端调用逻辑
```php
$settleComponent = new UserSettleComponent();

$vipSettleDecorator = new VipSettleDecorator();
$hightRateSettleDecorator = new HightRateSettleDecorator;


// 循环封装进去，这样子可以决定执行顺序
$vipSettleDecorator->setComponent($settleComponent);
$hightRateSettleDecorator->setComponent($vipSettleDecorator);

$hightRateSettleDecorator->settle($user,$order);
```
